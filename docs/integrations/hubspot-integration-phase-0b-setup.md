## Step 1 — Create 18 custom properties

**Path:** Settings (gear icon, top-right) → Properties (under Data Management)

For each property below, follow the same pattern:
1. Click "Create property"
2. Choose the object (Contact / Ticket / Deal / Company) per the table
3. Click "Next"
4. Paste the **Label** from the table — HubSpot will auto-suggest the internal name. **Verify the suggested internal name matches the table**; if not, override manually.
5. Choose the field type
6. **Group:** see Group recommendation below
7. Add description (optional but recommended)
8. For enum types, enter the options exactly as listed
9. Click "Create"

### Group recommendation

HubSpot organizes properties into Groups for visual grouping in the property editor. Two reasonable choices:

- **Default groups** (simplest): `Contact information` / `Ticket information` / `Deal information` / `Company information`. Works fine — our properties are clearly identifiable by their prefixes (`launchpad_*`, `rejig_*`, `onboarding_*`).
- **Custom group "LaunchPad Integration"** (cleaner long-term): create a custom group on each object and put all the integration-related properties there. Makes property management easier later when there are 30+ properties on Contact.

**If you've already created the 4 Contact properties in the default `Contact information` group:** leave them there. Don't bother moving. For new objects (Ticket, Deal, Company), you can choose either approach — I'd lean **default group for now** (keeps it consistent with what you already did). We can re-group later if it gets messy.

For Tickets specifically: yes, `Ticket information` is the default group and is fine. All 7 ticket properties go there.

## Step 2 — Pipeline Rules (SKIP for v1)

**Decision (2026-05-13): leave all 4 toggles OFF.** Pipeline Rules conflict with LaunchPad's intended push pattern (skipping stages, BI-driven healing moves backward) and don't fully constrain CSMs anyway.

**Path:** Settings → Tickets → Pipelines → Customer Journey Stages → Pipeline Rules tab — leave everything OFF.

### Why we're skipping

LaunchPad's pushes need to skip stages (Pre-Onboarding → Onboarding Scheduled jumps over old Intake/Design/Approval stages) and move backward (BI heals Critical → Active when customer recovers). "No skipping" and "no backward" rules could block LaunchPad if HubSpot enforces them on API writes too. Not worth the risk.

### Lock-down mechanism instead

1. **Trust:** tell Mario/Luis "BI manages stages; please don't move them. Add a Note explaining why if you must."
2. **Audit:** every CSM-driven stage change is logged via the `ticket.propertyChange:hs_pipeline_stage` webhook into LaunchPad's `hubspot_inbound_events` table. Admin reviews weekly.
3. **v2 (if needed):** LaunchPad webhook detects CSM-driven stage changes that conflict with BI state, posts a Note + reverts. Build only if data shows CSMs are abusing.

## Step 3 — Deal stage gate for Core Subscription ID

**Path:** Settings → Objects → Deals → Pipelines → (your primary deals pipeline) → click into the **Closed won** stage → "Required properties" section

Add: **Stripe Subscription ID — Core** (internal name: `stripe_payment_id`)

### Original Workflow design (preserved for reference when we pick this back up)

Below is the design we documented; keep as reference. **Do not build these yet.**

**Path:** Automations → Workflows → Create workflow → From scratch → Contact-based

Create the following workflows, but **leave them OFF until cutover**:

### Workflow A: "Meeting outcome → Completed"

- **Trigger:** Contact has associated meeting where `Meeting outcome` = `Completed`
- **Actions:**
  - Set associated Ticket's `Ticket status` (= `hs_pipeline_stage`) to **Active** (the renamed Healthy stage)
  - Set associated Ticket's `onboarding_no_show_count` to `0`

> **Note:** Don't try to set `onboarding_completion_state` — that property was dropped in the 2026-05-13 lean-property audit (see Step 1 line 60). The Meeting record's native `hs_meeting_outcome` already holds this state; duplicating it on the Ticket creates two writers and drift risk.

### Workflow B: "Meeting outcome → No-show"

- **Trigger:** Contact has associated meeting where `Meeting outcome` = `No-show`
- **Actions:**
  - Set associated Ticket's `Ticket status` to **Pre-Onboarding**
  - Increment `onboarding_no_show_count` by 1
  - Delay 24h → If still in Pre-Onboarding → Send email "We missed you" (template)
  - Delay 48h (T+72h total) → If still in Pre-Onboarding → Send email "Reminder to reschedule"
  - Delay 4 days (T+7d total) → If still in Pre-Onboarding → Send final email + set Ticket status to **Watch** with `rejig_attention_reason` = `no_show_no_rebook`

### Workflow C: "Meeting outcome → Partial"

- **Trigger:** Contact has associated meeting where `Meeting outcome` = `Partial`
- **Actions:**
  - Set associated Ticket's `Ticket status` to **Pre-Onboarding** (re-enters pre-onboarding so CSM sees it needs follow-up; the Meeting record's `hs_meeting_outcome = Partial` is the source-of-truth for "why")
  - Delay 48h → If still in Pre-Onboarding → Send email "Let's finish setting up"
  - Delay 5 days (T+7d total) → If still in Pre-Onboarding → Send email + set Ticket status to **Watch** with `rejig_attention_reason` = `partial_no_completion`

### Workflow D: "Meeting outcome → Cancelled"

- **Trigger:** Contact has associated meeting where `Meeting outcome` = `Cancelled`
- **Actions:**
  - Set associated Ticket's `Ticket status` to **Watch** (immediate)
  - Set `rejig_attention_reason` = `customer_cancelled_onboarding`
  - No emails (CSM should personally reach out)

### Workflow E: "Meeting outcome → Rescheduled"

- **Trigger:** Contact has associated meeting where `Meeting outcome` = `Rescheduled`
- **Actions:** none on the ticket (stay in current stage); date updates automatically from Meeting record.

**Note:** "Partial" likely isn't in HubSpot's default Meeting outcome enum. You may need to **add a custom option** to the `hs_meeting_outcome` property: Settings → Properties → Meeting properties → Meeting outcome → Edit options → add `Partial`.

**Email templates needed (Workflows B and C use these):**
- "We missed you" (no-show, 1st)
- "Reminder to reschedule" (no-show, 2nd)
- "Final reminder to reschedule" (no-show, 3rd)
- "Let's finish setting up" (partial, 1st)
- "Final partial follow-up" (partial, 2nd)

Create as HubSpot email templates (Marketing → Email → Templates) or write inline in the Workflow actions. Either works.
